# Title Heading

## Accessibility

### ARIA Attributes

The `ds-title-heading` component renders native heading elements (`h1`–`h6`), which inherently carry the `heading` role and appropriate `aria-level`. No additional ARIA attributes are required for basic usage.

Proper heading hierarchy is critical for accessibility:

### Keyboard Navigation

Heading elements are not interactive by default, so no keyboard navigation is needed. However, screen reader users rely on headings for navigation:

### Screen Reader Support

- The component uses native heading elements, which are universally supported by all screen readers.
- Screen readers announce the heading level automatically (e.g., "Heading level 2, Welcome").
- Visually hidden headings (`isVisuallyHidden: true`) are still announced — this is the intended behavior for providing structural navigation without visual clutter.
- The `introText` is rendered as a `div` with `innerHTML`, so ensure its content is accessible (no complex interactive elements).

### Best Practices

- **Always set `htmlTag` based on the document outline**, not visual design. Use `styleClass` independently for visual styling.
- **Never skip heading levels** in the document structure. If you need an `h2` that looks like an `h1`, set `htmlTag: TitleHeading.H2` and `styleClass: TitleHeading.H1`.
- **Use `isVisuallyHidden`** for section headings that screen reader users need but shouldn't be displayed (e.g., main content area landmark headings).
- **Keep `introText` simple** — since it uses `innerHTML`, avoid embedding complex interactive content. Plain text and basic formatting (`<br>`, `<strong>`, `<em>`) are appropriate.
- **Avoid empty headings** — always provide meaningful `title` text. Empty headings confuse screen reader navigation.
