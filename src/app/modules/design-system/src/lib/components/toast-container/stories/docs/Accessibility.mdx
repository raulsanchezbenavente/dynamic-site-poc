# Toast Container

## Accessibility

### ARIA Attributes

The `toast-container` itself does not add ARIA attributes — it dynamically renders `<toast>` components which handle their own accessibility. See the [Toast Accessibility](?path=/docs/atoms-toast-toast--02-accessibility) documentation for details on `role="alert"`, icon labels, and close button labeling.

### Screen Reader Support

- When a toast is inserted into a target container via `ToastService.show()`, the `role="alert"` on the toast element triggers an immediate announcement.
- The `data-toast-slot="panel"` attribute changes only visual positioning (sticky vs absolute) — it does not affect screen reader behavior.
- Avoid spawning multiple toasts in the same target simultaneously. Sequential toasts should wait for the previous one to be dismissed.

### Best Practices

- Use **one `<toast-container />`** per view/page. Multiple instances may cause duplicate toast rendering.
- Always assign a unique `id` to each target container — collisions can cause toasts to appear in unexpected locations.
- For panel-level workflows (e.g., account settings, profile editors), use `data-toast-slot="panel"` so the toast sticks to the top of the viewport while the user scrolls through long forms.
- For page-level notifications without panels, omit `data-toast-slot` and ensure the target container has `position: relative`.
