### Screen Reader Support

- **Appearance**: When a toast appears, the `role="alert"` triggers an immediate announcement. The status icon label (e.g., "Success") is read first, followed by the message.
- **Dismissal**: When `autohide` is enabled, the toast disappears after the configured `delay` (default: 6000ms). Screen readers do not announce the removal, which is expected behavior for transient notifications.
- **Multiple toasts**: Avoid showing multiple toasts simultaneously. Rapid sequential announcements can overwhelm screen reader users. Use the `hidden` output event to manage toast lifecycle.

### Best Practices

- Always pair the `status` type with a message that matches its severity — don't use `SUCCESS` for error-like messages.
- Set `autohide: false` for error toasts that require user attention, ensuring they aren't dismissed before being read.
- For `delay`, use a minimum of 5000ms to give screen reader users sufficient time to perceive, navigate to, and read the message.
- Ensure toast messages are concise (under 100 characters) so screen readers can announce them quickly.
- When using `data-toast-slot="panel"` in a `toast-container` context, the sticky positioning ensures the toast remains visible while scrolling — this is the recommended pattern for panel-level notifications.
